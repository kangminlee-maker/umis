## 개요

UMIS v7.2는 **2가지 추정 방법론**을 제공합니다:

| 방법론 | 속도 | 정확도 | 적합 상황 |
|--------|------|--------|----------|
| **Guestimation** | ⚡ 5-30분 | ±50% | 초기 탐색, 빠른 판단 |
| **Domain Reasoner** | 🔬 1-4시간 | ±30% | 정밀 분석, 투자 심사 |
| **Hybrid (Auto)** | 🔄 15분-4시간 | ±30-50% | 모든 상황 (자동 적응) |

---

## 2가지 방법론

### 1️⃣ UMIS Guestimation

**철학**: "보이지 않는 것을 보이게. 정답보다 과정이 중요"

**방식**:
- Fermi 4원칙 (모형, 분해, 제약, 자릿수)
- 8가지 데이터 출처 자유 조합

**예시**:
```
질문: "음악 스트리밍 시장 규모?"

분해:
- 한국 인구 5,200만
- 스마트폰 보급률 95% → 4,940만
- 음악 앱 사용률 60% → 2,964만
- 유료 전환 20% → 593만
- ARPU 월 1만원 → 연 12만원
- 시장 규모: 593만 × 12만 = 7,116억

결과: 약 7,000억 (±50%)
소요: 10분
```

**언제 사용?**:
- ✅ 초기 탐색 (목표 불명확)
- ✅ 빠른 기회 우선순위
- ✅ 성숙 시장 (데이터 풍부)
- ❌ 규제 산업
- ❌ 투자 심사

---

### 2️⃣ Domain-Centric Reasoner

**철학**: "RAG는 근거 제공자. 규제/물리가 최우선"

**방식**:
- 10가지 신호 우선순위 (s3→s8→s6→s10→s2→...)
- RAG 중심 (s2, s9, s10)
- 6단계 파이프라인

**예시**:
```
질문: "시니어 케어 로봇 시장 규모 (2030)?"

신호 처리:
1. s10 (KPI 정의): "케어 로봇 시장 = ?"
2. s3 (규제): 의료기기법, 개인정보보호법
3. s2 (RAG Consensus): 직접 데이터 없음
4. s9 (Case Analogies): 일본 $500M → 전이 보정
5. s4 (Behavioral): Should 5,000억 vs Will 2,850억

결과:
- 점추정: 2,850억 (±30%)
- Should: 5,000억 (사회적 필요)
- Will: 2,850억 (채택 장벽)
- Gap: 43% (기술 거부감, 가격 부담)
- 증거표: 3개 독립 출처
- 검증 로그: ✅✅⚠️✅

소요: 2시간
```

**언제 사용?**:
- ✅ 신규 시장 (사례 전이)
- ✅ 규제 산업 (s3 Laws)
- ✅ 투자 심사 (증거표)
- ✅ 정밀 분석 (±30%)
- ❌ 긴급 판단 (< 1시간)
- ❌ 간단한 질문

## 사용법

### 방법 1: @ 명령어 (권장)

#### 자동 판단 (Hybrid)
```bash
@auto 국내 OTT 시장 규모
```
→ Guardian이 자동으로 최적 방법론 선택

#### 명시적 Guestimation
```bash
@guestimate 음악 스트리밍 시장 규모
```
→ 빠른 추정 (5-30분)

#### 명시적 Domain Reasoner
```bash
@reasoner 시니어 케어 로봇 시장 규모
```
→ 정밀 분석 (1-4시간)

### 방법 2: Agent 지정

```bash
# Explorer + Guestimation
@Explorer guestimate 구독 모델 기회 크기

# Quantifier + Domain Reasoner
@Quantifier reasoner 배달 플랫폼 수수료율

# Validator + Domain Reasoner (KPI 검증)
@Validator reasoner 전환율 정의 확인
```

## 하이브리드 전략

### 🔄 자동 전환 플로우

```
Step 1: Phase 1 실행 (항상)
  → UMIS Guestimation (5-30분)
  → 결과: 추정값, 범위, 신뢰도

Step 2: Guardian 평가
  → 5가지 트리거 확인
  → Phase 2 필요 여부 판단

Step 3: Phase 2 실행 (조건부)
  → Domain-Centric Reasoner (1-4시간)
  → 결과: 증거표, Should/Will, 검증 로그
```

### 🎯 5가지 전환 트리거

| 트리거 | 조건 | 우선순위 | 자동 실행 |
|--------|------|----------|-----------|
| 1. 낮은 신뢰도 | < 50% | high | ❌ (확인) |
| 2. 넓은 범위 | > ±75% | high | ❌ (확인) |
| 3. 큰 기회 | > 1,000억 | medium | ❌ (제안) |
| 4. 규제 산업 | 의료/금융/교육 | required | ✅ (필수) |
| 5. 신규 시장 | 직접 데이터 부족 | medium | ❌ (제안) |

**우선순위**: required > high > medium > low

**결정 로직**:
- `required` → 자동 실행
- `high` → 사용자 확인 후 실행
- `medium` → 제안 (선택)
- `low` → Phase 1 완료

## 출력 비교

### Guestimation 출력

```yaml
결과:
  추정값: 8.5%
  범위: 7-10%
  신뢰도: 70%

문서화 (7개 섹션):
  [1] 추정 필요 이유
  [2] 사용한 데이터 (SRC_ID)
  [3] 추정 논리 (단계별)
  [4] 신뢰도 평가
  [5] 검증 방법
  [6] 대체 접근법
  [7] 사용 위치

ID:
  EST_001: 8.5%
```

### Domain Reasoner 출력

```yaml
결과:
  점추정: 8.5%
  합의 범위: 6-12%
  신뢰도: Medium-High (70%)

Should vs Will:
  Should (규범): 7.5% (공정성)
  Will (현실): 9.5% (시장 지배력)
  Gap: 26% (독과점 영향)

증거표:
  - SRC_001: UMIS RAG "배민 6-12%"
  - SRC_002: 검색 "배민 수수료"
  - 독립성: ✅

검증 로그:
  - 단위 일관성: ✅
  - 규제 준수: ✅
  - 비교사례 합의: ✅
  - Should/Will 분리: ✅

민감도:
  - 플랫폼 집중도: ★★★★★
  - 정부 규제: ★★★★☆
  - 배달원 수급: ★★★☆☆

다음 행동:
  1. 쿠팡이츠 수수료 공시 확인
  2. 음식점 인터뷰
  3. 업종별 차별 수수료 조사
```


## 📊 실전 팁

### Tip 1: 신뢰도 확인

Phase 1 결과의 신뢰도를 확인하세요:
- **> 70%**: Guestimation 충분
- **50-70%**: 상황에 따라
- **< 50%**: Domain Reasoner 권장

### Tip 2: 범위 폭 확인

범위가 너무 넓으면 (±75% 이상) 정밀 분석 필요:
- **1,000억-2,000억** (±50%) → OK
- **500억-5,000억** (±400%) → Phase 2 필요

### Tip 3: 기회 크기 확인

- **< 100억**: Guestimation만으로 충분
- **100-1,000억**: 상황에 따라
- **> 1,000억**: Domain Reasoner 권장

### Tip 4: 규제 산업은 무조건 Phase 2

의료, 금융, 교육 등 규제 산업은:
- s3 (Laws/Ethics/Physics) 검증 필수
- 자동으로 Phase 2 실행

### Tip 5: Should vs Will 활용

**Should (규범적)**:
- 정책 권고: "적정 수수료는 7%"
- 목표 설정: "잠재 시장 5,000억"

**Will (현실적)**:
- 실제 예측: "시장 지배력 → 9.5%"
- 매출 전망: "채택 장벽 → 2,850억"

**Gap 활용**:
- 개선 기회: "시장 경쟁 촉진 → Gap 축소"
- 현실 수용: "Gap 50% = 구조적 한계"
